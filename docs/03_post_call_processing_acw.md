# Блок 3: Post-Call Processing + ACW Auto-fill

> **Пріоритет**: P1 ⭐ — Перший для пілотного впровадження
> **Swimlane**: DATA PIPELINE → ACW
> **Складність**: High (Pipeline) + Low-Medium (ACW UI)
> **Об'єднує**: STT Pipeline, LLM Post-Processing, ACW Auto-fill та Validation

---

## Роль у архітектурі ToBe

Data Pipeline — серцевина постобробки. Cisco CUBE форкує аудіо через SIPREC і передає RTP-потік до SRS Bridge **в реальному часі під час дзвінка**. SRS Bridge буферизує потік у пам'яті, а після отримання SIP BYE (завершення дзвінка) конвертує аудіо і відправляє на STT для пакетної обробки. STT транскрибує запис, LLM генерує структурований аналіз: резюме, класифікація теми, витяг ентітій, оцінка sentiment. Результат автоматично заповнює ACW-форму оператора в CRM та зберігається в базі даних для аналітики.

**Чому P1**: найвищий ROI серед усіх компонентів. Економія на ACW ~$487K/міс при вартості впровадження ~$12–38K/міс. Payback period — менше місяця після запуску.

## Технічний опис

### Протоколи та транспорт

Cisco CUBE виступає як SIPREC SRC (Session Recording Client) і форкує RTP G.711 µ-law 8kHz потік **в реальному часі під час активного дзвінка** до SRS Bridge (Session Recording Server). SRS Bridge (мікросервіс Go/Rust) буферизує RTP-фрейми в пам'яті. Після отримання SIP BYE (сигнал завершення дзвінка) — збирає повний аудіофайл, конвертує у FLAC/WAV і запускає STT-обробку. **"Batch" тут означає режим STT-запиту (весь файл одразу після дзвінка), а не затримку передачі аудіо.** Це знижує вартість STT і спрощує архітектуру порівняно з real-time streaming режимом Блоку 4.

| Протокол | Провайдер | Latency (batch mode) |
|----------|-----------|---------------------|
| HTTP POST batch | OpenAI Whisper API | ~5–15 сек/хв аудіо |
| gRPC / HTTP batch | Google STT Chirp 3 | ~100–200 мс |
| HTTP POST batch | faster-whisper self-hosted | ~1–5 сек/хв аудіо |

### Latency вимоги

| Стадія | Цільовий час |
|--------|-------------|
| STT транскрипція (batch) | < 30 сек після кінця дзвінка |
| LLM постобробка | < 60 сек (синхронний запит) |
| Доступність у CRM (ACW) | < 90 сек після кінця дзвінка |
| Deep analytics (Ender Turing) | < 10 хв |

## Варіанти реалізації

### Варіант 1 ⭐ Рекомендований: OpenAI Whisper + GPT-4o

> **Найдешевший хмарний STT + найвища якість LLM.** OpenAI Whisper — $0.006/хв (у 2.7x дешевше Google Chirp 3). GPT-4o — найсильніший LLM для аналізу розмов. Єдиний вендор, один контракт.

#### Архітектура

```
Cisco CUBE
  │ SIPREC RTP G.711 µ-law 8kHz (real-time, під час дзвінка)
  ▼
SIPREC SRS Bridge (Go/Rust мікросервіс)
  - Приймає RTP-фрейми в реальному часі
  - Буферизує аудіо в пам'яті
  - Після SIP BYE: конвертує G.711 → MP3/WAV
  - Завантажує до хмарного сховища (S3 / GCS)
  │ HTTP POST (після завершення дзвінка)
  ▼
OpenAI Whisper API (Whisper large-v3)
  - STT uk-UA з підтримкою speaker diarization
  - Timestamps на рівні слів
  │ Транскрипт
  ▼
OpenAI GPT-4o API (синхронний, для ACW < 90 сек)
  - Input: транскрипт + system prompt з правилами НП
  - Output: summary, topic, subtopic, sentiment, entities, disposition_code
  │ Structured JSON → черга
  ▼
CRM Connector
  - POST /api/v1/calls/{call_id}/acw → власна CRM НП
  │
  ▼
BigQuery / власна БД (транскрипти + аналітика → Ender Turing)
```

#### Переваги в контексті НП

**1. Найдешевший STT серед хмарних варіантів**
OpenAI Whisper API — $0.006/хв проти $0.016/хв Google Chirp 3. При 75,000 хв/день це **~$13,500/міс vs ~$36,000/міс** — економія $22,500/міс лише на STT.

**2. GPT-4o — найвища якість LLM-аналізу**
Найкраще розуміє контекст, тон, нюанси скарг і ескалацій серед усіх доступних моделей. Підходить і для ACW-заповнення, і для QA-аналізу в Блоці 5 — один LLM для двох задач.

**3. Єдиний вендор — спрощена операційна модель**
STT і LLM від OpenAI — один контракт, одна команда підтримки, єдиний білінг.

**4. Whisper верифікований для uk-UA**
Whisper large-v3 підтримує uk-UA, включаючи регіональні акценти та змішану мову. Speaker diarization доступний через `timestamp_granularities`.

#### Ризики в контексті НП

| Ризик | Рівень | Деталі |
|---|---|---|
| **Вся розробка — на команді НП** | **Високий** | OpenAI — сирий API без готової contact center інтеграції. SRS Bridge, pipeline оркестрація, CRM connector, error handling, retry логіка, моніторинг — все пише команда НП. OpenAI Support не допомагає з UCCE-специфікою. Додатково: +4–8 тижнів розробки та підтримка після launch |
| Vendor lock-in OpenAI | Середній | STT + LLM в одного вендора. Зміна будь-якого компонента = рефакторинг pipeline |
| Відсутність публічного uk-UA WER | Середній | Немає порівняльних даних Whisper vs Chirp 3 для uk-UA. Потрібен POC на реальних записах НП |
| GPT-4o sync вартість | Низький | Синхронний GPT-4o дорожче ніж Batch (~2x). Але при обсягах НП різниця ~$750/міс |

> ⚠️ **Ключова відмінність від Google**: Google Cloud (Варіант 2) надає managed сервіси — Cloud Run, Pub/Sub, GCS triggers, BigQuery — з готовою документацією і підтримкою. OpenAI дає лише два HTTP endpoint'и (Whisper + GPT-4o). Весь "клей" між компонентами — відповідальність команди розробки НП.

**Коли обирати для НП:**
Якщо команда розробки НП має досвід побудови data pipeline і готова взяти на себе повну відповідальність за інтеграцію. Найнижча cloud-вартість (~$17,250/міс) і найвища якість LLM компенсують вищі one-time витрати на розробку. Потрібен POC для підтвердження WER на реальних записах НП.

> **Доцільно якщо**: НП обирає Google як основний cloud-вендор для всіх AI-компонентів (Блоки 3, 4, 5 — в одному GCP проекті).

#### Архітектура

```

  │ SIPREC RTP G.711 µ-law 8kHz (real-time, під час дзвінка)
  ▼

  - Після SIP BYE: конвертує G.711 → FLAC, завантажує до GCS
  │ GCS event trigger (після завершення дзвінка)
  ▼
Cloud Run Job
  - Google STT Chirp 3 Batch API (uk-UA, speaker diarization)

  │ Транскрипт
```

#### Переваги в контексті НП

**3. Найзріліший enterprise-стек**
GCS + BigQuery + Cloud Run + Pub/Sub — перевірена архітектура з максимальним uptime SLA.

#### Ризики в контексті НП

| Ризик | Рівень | Деталі |
|---|---|---|
| **Найвища STT вартість** | **Критичний** | ~$36,000/міс — у 2.7x дорожче OpenAI Whisper без підтвердженої переваги в якості для uk-UA |

| Gemini Flash якість нижча ніж GPT-4o | Середній | Для простих задач — достатньо. Для QA і compliance — поступається GPT-4o |

**Коли обирати для НП:**
Тільки якщо НП стратегічно будує Google-first екосистему і єдина vendor-угода важливіша за оптимізацію вартості. Переплата $22,500/міс на STT порівняно з Варіантом 1 виправдана лише нативною GCP-інтеграцією.

### Варіант 3: Self-hosted Whisper + Llama 3.1

> **Застосування**: виключно при законодавчих вимогах до локалізації даних. Не рекомендується з економічних міркувань при поточному масштабі.

#### Архітектура

```
Cisco CUBE
  │ SIPREC RTP G.711 µ-law 8kHz (real-time, під час дзвінка)
  ▼
SIPREC SRS Bridge (on-premise, Go/Rust мікросервіс)
  - Приймає RTP-фрейми в реальному часі
  - Буферизує аудіо в пам'яті
  - Після SIP BYE: конвертує G.711 → WAV
  - Зберігає у внутрішнє сховище НП (NFS/Ceph)
  │ (дані не покидають інфраструктуру НП)
  ▼
faster-whisper large-v3 (GPU кластер on-prem, 8–12x A100/H100)
  - STT uk-UA batch — overnight або near-real-time черга
  - LoRA fine-tune на даних НП (ТТН-номери, відділення, жаргон)
  │ Транскрипт
  ▼
Llama 3.1 70B / vLLM (GPU кластер on-prem)
  - LLM постобробка — summary, topic, sentiment, entities
  │ Structured JSON
  ▼
CRM Connector (on-prem сервіс)
  - POST /api/v1/calls/{call_id}/acw → власна CRM НП
  │
  ▼
Внутрішня БД НП (аналітика без виходу назовні)
```

#### Переваги в контексті НП

**1. Повна суверенність даних**
Жодне аудіо і жоден транскрипт не покидає інфраструктуру НП. Відповідає найсуворішим compliance-вимогам і законодавству про персональні дані.

**2. Нульова змінна вартість per-call після CAPEX**
При масштабуванні понад 80–100K дзвінків/день — єдиний варіант де вартість не росте лінійно.

#### Ризики в контексті НП

| Ризик | Рівень | Деталі |
|---|---|---|
| CAPEX $400K–$800K | **Критичний** | GPU кластер 8–12x A100/H100. Окупність 2+ роки при поточних 30K дзвінків/день |
| Постійна MLOps-команда | **Високий** | GPU моніторинг, оновлення моделей, fine-tuning — окрема ставка або підрядник $10–20K/міс |
| uk-UA fine-tuning обов'язковий | Середній | Специфіка НП: ТТН-номери, назви відділень, jargon — потрібен LoRA fine-tune на 10K+ прикладах |
| Немає SLA від вендора | Середній | Відповідальність за uptime — повністю на команді НП |

**Коли обирати для НП:**
Тільки при наявності законодавчих вимог до локалізації даних АБО при масштабі 100K+ дзвінків/день де cloud-вартість перевищує CAPEX. При поточних 30K/день — не рекомендується.

## Інтерфейси взаємодії системи

| Напрямок | Від | До | Протокол | Дані |

- `PATCH /api/v1/operators/{id}/stats` — оновлення статистики оператора

**Payload до CRM містить:** call_id, operator_id, queue, duration_sec, transcript_uri, summary, topic, subtopic, sentiment (overall / agent / client), checklist_completed, checklist_missed, entities (parcel_id, department, city), disposition_code, processing_latency_sec.

## Складність впровадження

**Рівень: High**

| Компонент | Відповідальний | Складність | Обґрунтування |
|-----------|--------------|------------|---------------|
| SIPREC SRS Bridge | НП / підрядник | High | Мікросервіс Go/Rust, тестування з Cisco CUBE, networking |
| STT інтеграція | НП / підрядник | Medium | Добре задокументовані API, тонке налаштування uk-UA |
| LLM Pipeline (GPT-4o) | НП / підрядник | Medium | Prompt engineering + валідація якості на реальних даних НП |
| CRM інтеграція | НП внутрішня команда | Medium | Залежить від існуючого CRM API — ключовий ризик |
| Оркестрація (черга + тригери) | НП / підрядник | Medium | Стандартна хмарна архітектура |

**Ключові ризики при 30K дзвінків/день:**
- **Якість STT для uk-UA**: суржик, регіональні акценти, шум знижують WER. Обов'язковий POC з реальними записами НП до production
- **Prompt engineering**: якість GPT-4o аналізу залежить від промптів. Потрібна ітерація на 200–500 реальних дзвінках НП
- **CRM API rate limiting**: 30K запитів/день до власної CRM — потрібна черга або batch insert
- **SIPREC bridge latency**: мережева архітектура між CUBE і SRS Bridge впливає на якість аудіо і затримку

## ACW Auto-fill + Validation

> Pipeline (STT + LLM) генерує дані → ACW Auto-fill доставляє їх оператору → оператор переглядає і підтверджує. Це два шари одного процесу, а не окремі компоненти.

### Чому ACW має найвищий ROI серед усіх компонентів

| Метрика | Ручний ACW | AI-асистований ACW | Різниця |
|---------|-----------|-------------------|---------|
| Час/дзвінок | ~4 хв | ~45 сек (review) | **-75%** |
| Оператор-годин/день | 2,000 год | 375 год | -1,625 год |
| Місячна вартість (при $10/год) | ~$600,000 | ~$112,500 | **-$487,500/міс** |
| Річна економія | — | — | **~$5.85M/рік** |

> При консервативній оцінці $5/год — економія ~$2.9M/рік. Payback period < 1 місяця від запуску.

### UX-процес: як оператор бачить ACW

Після завершення дзвінка pipeline обробляє аудіо протягом < 90 сек і надсилає payload до CRM. Оператор бачить вже заповнену ACW форму: тема, підтема, резюме, disposition, entities. Переглядає (~45 сек), натискає "Підтвердити" або коригує окремі поля. CRM зберігає результат → Analytics (BigQuery / Ender Turing).

### ACW payload (що передається в CRM)

| Поле | Що це | Приклад |
|---|---|---|
| `topic` / `subtopic` | Тема і підтема з дерева тематик НП | "Відстеження" / "Затримка доставки" |
| `summary` | Резюме дзвінка (1–3 речення) | "Клієнт запитав статус посилки..." |
| `disposition_code` | Код завершення | `TRACKING_DELAY` |
| `entities` | Ключові дані з розмови | parcel_id, department, promised_date |
| `checklist` | Виконані/невиконані пункти | greeting ✅, offer_promo ❌ |
| `sentiment` | Тональність (overall/agent/client) | neutral / positive / slightly_negative |
| `ai_confidence` | Впевненість моделі | 0.91 |

### Фази впровадження ACW

**Фаза 1 — "Тихий пілот" (тижні 1–4)**
Pipeline генерує ACW payload для всіх дзвінків, але операторам **не показує**. QA-команда порівнює AI-пропозиції з ручним заповненням.
- Ціль: точність теми > 85%, disposition > 70%

**Фаза 2 — "Показ без обов'язкового прийняття" (тижні 5–8)**
Pre-filled форма показується оператору. Оператор може прийняти або змінити.
- Вимірювати: % прийнято без змін, середній час ACW

**Фаза 3 — Production**
- Критерій переходу: точність теми > 88%, % прийнятих без змін > 70%, середній ACW < 60 сек

### Ризики ACW

- **Якість промптів**: потрібне ретельне налаштування під дерево тематик НП і термінологію
- **Прийняття операторами**: оператори мають довіряти AI-пропозиціям. Потрібен onboarding і навчання
- **CRM API**: власна CRM НП може не мати готового ACW endpoint — потребує розробки з боку CRM-команди
