# Блок 3: Автоматичне заповнення картки після дзвінка (Post-Call ACW)

> **Пріоритет**: P1 ⭐ — перший для впровадження, найвищий ROI
> **Залежить від**: Блок 1 (копіювання аудіо — обов'язковий)
> **Передає дані до**: РМ КЦ (картка оператора), Блок 5 (аналітика)

## Що це і навіщо

Після кожного дзвінка оператор вручну заповнює картку: тема звернення, резюме, код завершення — це займає ~4 хвилини на дзвінок. Блок 3 автоматично заповнює цю картку протягом 15 секунд після завершення дзвінка: система слухає розмову в реальному часі, розпізнає текст, аналізує його через AI і відправляє готові дані в РМ КЦ (CRM). Оператор лише переглядає результат (~45 сек) і натискає "Підтвердити". Мета — скоротити час ACW на 75% і звільнити оператора для наступного дзвінка.

## Рекомендоване рішення ⭐

Рекомендується Варіант 1: Google STT Chirp 3 + GPT-4o — єдиний хмарний варіант, що надійно вкладається у вимогу ACW < 15 секунд завдяки розпізнаванню мови в реальному часі (під час дзвінка), а не після його завершення. GPT-4o забезпечує найвищу якість аналізу тексту.

## Варіанти рішень

### Варіант 1: Google STT + GPT-4o ⭐

**Що це:** Система слухає аудіо в реальному часі і розпізнає текст ще під час розмови (gRPC streaming). Коли дзвінок завершується — транскрипт вже повністю готовий. GPT-4o аналізує текст за 3–5 секунд і відправляє заповнену картку в РМ КЦ (CRM). Загальна затримка: 5–8 секунд після завершення дзвінка — з запасом вкладається у 15 секунд.

**Переваги:**
1. Єдиний хмарний варіант, що надійно відповідає вимозі ACW < 15 секунд
2. GPT-4o — найвища якість LLM-аналізу: найкраще розуміє контекст, тон, нюанси скарг і ескалацій
3. Google STT — managed enterprise сервіс з готовою документацією і підтримкою
4. Спільний STT-потік з Блоком 4 (якщо впроваджуються разом) — STT оплачується один раз, не двічі

**Ризики:**
1. Найвища STT-вартість (~$36,000/міс) — обов'язкова ціна за < 15 сек ACW, batch-варіанти дешевші, але не вкладаються у вимогу
2. GPT-4o — сирий API, вся розробка pipeline (промпти, обробка помилок, retry-логіка) на команді НП або підрядника
3. Два контракти (Google + OpenAI) — незначна операційна складність
4. Відсутність публічних даних якості розпізнавання для uk-UA — потрібен тест на реальних записах НП

**Вартість:** одноразово ~$35–65K (власна розробка сервера SRS Bridge) | щомісяця ~$39,750

> ⚠️ Сервер обробки аудіо (SRS Bridge) — власна розробка команди НП або підрядника
> Vendor API: Google STT (Google відповідає) + OpenAI GPT-4o (OpenAI відповідає)

---

### Варіант 2: Google STT + Gemini 2.0 Flash

**Що це:** Та сама архітектура, що і Варіант 1 (Google STT streaming під час дзвінка), але замість GPT-4o для аналізу тексту використовується Gemini 2.0 Flash — значно дешевший LLM від Google. Повністю Google-стек: STT + LLM + аналітика (BigQuery) + інфраструктура (Cloud Run, Pub/Sub) — один контракт.

**Переваги:**
1. Єдиний Google-вендор — один контракт, одна команда підтримки Google
2. Gemini 2.0 Flash — найдешевший LLM (~$43/міс замість ~$2,250/міс для GPT-4o)
3. Нативна GCP-екосистема: менше custom коду, готові тригери і конектори знижують one-time розробку

**Ризики:**
1. Найвища STT-вартість (~$36,000/міс) — аналогічно Варіанту 1, неминуча ціна за streaming
2. Повний vendor lock-in Google — BigQuery + GCS + STT + Gemini = повна залежність від одного вендора
3. Gemini Flash якість нижча ніж GPT-4o — для простих задач (резюме, тема, disposition) достатньо, але для QA-аналізу і compliance поступається

**Вартість:** одноразово ~$35–65K (власна розробка) | щомісяця ~$37,543

---

### Варіант 3: OpenAI Whisper + GPT-4o ⚠️ (лише якщо ACW < 40 сек)

**Що це:** Whisper розпізнає текст лише після завершення дзвінка (batch-режим): спочатку потрібно зберегти весь аудіофайл, потім відправити на сервер, потім дочекатись результату. Для дзвінка 2–3 хв це займає 12–30 сек тільки на розпізнавання, плюс ще 3–5 сек на аналіз GPT-4o. Разом: 15–35 секунд після завершення дзвінка — вимогу < 15 сек не виконує надійно.

**Переваги:**
1. Найдешевший STT — ~$13,500/міс ($0.006/хв проти $0.016/хв Google, економія ~$22,500/міс)
2. Єдиний вендор OpenAI (Whisper + GPT-4o) — один контракт, один білінг

**Ризики:**
1. Не відповідає вимозі ACW < 15 сек — batch processing після BYE: 12–30 сек Whisper + 3–5 сек GPT-4o = 15–35 сек; непередбачувано при пікових навантаженнях
2. Вся розробка — на команді НП: OpenAI надає лише HTTP endpoint, SRS Bridge, pipeline, РМ КЦ connector, error handling — все пише команда НП
3. Відсутність публічних даних якості розпізнавання для uk-UA — потрібен тест на реальних записах НП

**Вартість:** одноразово ~$35–65K (власна розробка) | щомісяця ~$17,250

> ⚠️ Не відповідає вимозі ACW < 15 сек. Розглядати тільки якщо вимогу змінено до < 40 сек.

## Важливо: спільне використання з Блоком 4

Вимога ACW < 15 сек змушує Блок 3 слухати аудіо в реальному часі під час дзвінка — ту саму інфраструктуру потребує і Блок 4 (підказки оператору під час дзвінка). Це створює два сценарії впровадження:

**Комбінація A — Блок 4 через Google CCAI Connector (готовий продукт Google):**
Спільний потік аудіо неможливий — Google CCAI Connector не надає сирий транскрипт для Блоку 3. Блок 3 потребує власного окремого сервера (SRS Bridge). Google STT оплачується двічі: ~$72,000/міс замість $36,000/міс.

**Комбінація B — Блок 4 через власний SRS Bridge (той самий сервер):**
Один сервер (SRS Bridge) обслуговує обидва блоки одночасно: один Google STT потік — транскрипт читають і Блок 3 (ACW), і Блок 4 (підказки). STT оплачується один раз. One-time розробка: ~$100–115K разом проти ~$130K окремо (~$35K на Блок 3 + ~$95K на Блок 4).

> Якщо НП обирає ACW < 15 сек, вона вже будує 60–70% інфраструктури Блоку 4. Додати Real-time Copilot після цього — лише Agent Assist backend + WebSocket push + RМ КЦ widget. Паралельне впровадження скорочує сукупний бюджет і усуває необхідність переробляти сервер пізніше.

## Рекомендація для пілоту

1. Взяти 500–1,000 архівних записів дзвінків НП (WAV/MP3) і протестувати якість розпізнавання через Google STT Chirp 3 — перевірити точність на реальних записах (ТТН-номери, назви відділень, регіональні акценти)
2. Протестувати LLM-промпти на GPT-4o і Gemini Flash на 20–50 дзвінках з ручною перевіркою операторами — оцінити якість заповнення картки
3. Підключити до тестового ендпоінту РМ КЦ (CRM) — перевірити що API приймає автоматичне заповнення
4. Критерії успіху пілоту: точність розпізнавання теми > 85%, ACW доступний < 15 сек після завершення дзвінка, % прийнятих карток без змін > 70%
5. Після успішного пілоту — live streaming з 100 операторів, далі розширення на всіх 1,000
