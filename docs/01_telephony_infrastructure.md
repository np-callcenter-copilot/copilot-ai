# Блок 1: Cisco SIPREC + Медіа-форкінг

> **Пріоритет**: Базовий — фундамент для всіх інших блоків
> **Swimlane**: TELEPHONY
> **Складність**: Medium-High
> **Платформа**: Cisco UCCE 12.6(2) + CUCM 12.6 + CVP 12.6

---

## Роль у архітектурі ToBe

SIPREC + медіа-форкінг — фундамент усієї AI-архітектури. Без нього жоден інший блок не отримає аудіо для обробки.

**Завдання**: дублювати аудіо з кожного дзвінка до AI-систем **без будь-якого впливу на якість основного дзвінка** для оператора і клієнта.

**Механізм**: Cisco UCCE 12.6(2) підтримує **media-forking** — паралельну копію аудіопотоку за протоколом **SIPREC (RFC 7866)**. Cisco виступає як SRC (Session Recording Client), AI-система — як SRS (Session Recording Server). Один форкований потік може одночасно живити real-time Copilot (Блок 4) і batch Post-Call Pipeline (Блок 3).

## Варіанти підключення

### Варіант 1 ⭐ Рекомендований: Google CCAI Platform (нативна інтеграція Cisco + Google)

**Суть**: Cisco UCCE 12.6 має вбудовану підтримку Google CCAI Platform через **Cisco CCAI Connector** — окремий сервіс у UCCE-середовищі. Автоматично приймає аудіо від CUCM і передає до Google без розробки власного SRS.

#### Переваги в контексті НП

**1. Власний SRS-мікросервіс не потрібен — мінімальна вартість розробки**
$10–20K one-time (налаштування інженером) порівняно з $50–90K для варіантів з власним SRS. Найшвидший старт для пілоту.

**2. Нативна інтеграція з Cisco Finesse**
Підказки Google Agent Assist з'являються безпосередньо у Finesse — стандартному робочому столі операторів НП. Нульова крива адаптації: оператори вже там працюють.

**3. Офіційна підтримка Cisco + Google**
Єдиний варіант з офіційною спільною SLA і партнерською підтримкою. Найнижчий ризик при першому production-впровадженні масштабу НП.

**4. Перевірена платформа на enterprise масштабі**
Google CCAI використовують Vodafone, Deutsche Telekom та інші великі контакт-центри. Платформа тестована на тисячах concurrent сесій.

**5. Вбудована база знань і sentiment**
Google Agent Assist індексує документи НП (PDF/HTML). Sentiment analysis вбудований — автоматичний тригер ескалації при виявленні негативу без додаткових налаштувань.

#### Ризики в контексті НП

| Ризик | Рівень | Деталі |
|---|---|---|
| Vendor lock-in Google | Середній | Будь-яка заміна AI-провайдера = повна переробка архітектури |
| Підказки лише у Finesse | Середній | Якщо НП хоче підказки у власній CRM — потрібна додаткова WebSocket-інтеграція |
| Шаблонне розуміння Agent Assist | Середній | Відповіді по базі знань від поточної репліки. Не враховує контекст всієї розмови |
| Вартість Google STT real-time | Низький | ~$28,800–43,200/міс. Компенсується відсутністю вартості SRS-розробки порівняно з Варіантом 2 |

**Коли обирати для НП**: Google обраний як основна AI-платформа (Блок 3, 4, 5 також на Google) або пріоритет — мінімальна вартість і ризик першого запуску. Оптимальний варіант для пілоту.

### Варіант 2: OpenAI Realtime API (власний SRS-мікросервіс)

**Суть**: Власний SIPREC SRS-мікросервіс приймає аудіо від CUCM і передає до **OpenAI Realtime API** (GPT-4o). GPT-4o одночасно транскрибує і генерує підказки оператору в одному API-запиті. Аудіо передається у нативному форматі G.711 без конвертації.

#### Переваги в контексті НП

**1. Контекстне розуміння всієї розмови**
GPT-4o тримає повний контекст діалогу від першої репліки. Якщо клієнт третій раз звертається з тією ж проблемою — GPT-4o розуміє ескалацію. Google Agent Assist видасть стандартну статтю з бази знань.

**2. Нативна підтримка G.711 — не потрібна конвертація**
OpenAI Realtime API приймає G.711 µ-law — формат аудіо CUCM. SRS-мікросервіс простіший, немає трансформації потоку.

**3. Бізнес-логіка НП у system prompt**
Всі правила — компенсації, VIP-ескалація, процедури для затриманих посилок — описуються текстом. Зміна правил = редагування промпту, без переналаштування Dialogflow flows чи коду.

**4. Підказки у власній CRM НП (не лише Finesse)**
Через WebSocket підказки можна відображати у будь-якому інтерфейсі — власній CRM, Finesse overlay або окремому UI. Повна гнучкість.

#### Ризики в контексті НП

| Ризик | Рівень | Деталі |
|---|---|---|
| Найвища вартість | **Критичний** | ~$57,600–86,400/міс (gpt-realtime). У 1.5–2 рази дорожче Google навіть з новою моделлю |
| Потрібна розробка SRS + UI | Критичний | $50–90K one-time, 10–16 тижнів. Постійна підтримка команди після запуску |
| OpenAI — API, не продукт | Середній | Немає вендора що відповідає за інтеграцію з Cisco. OpenAI Support не допомагає з CUCM 12.6-специфікою |
| Ймовірнісний routing | Середній | GPT-4o може відповідати несподівано у рідкісних сценаріях. Для жорстких routing-правил НП — ризик |

**Коли обирати для НП**: після пілоту, якщо Google Agent Assist не покриває специфіку НП (складні ескалації, логістичний контекст) або НП будує власну CRM з AI-copilot поза Finesse.

## Складність впровадження

**Рівень: Medium-High**

| Компонент | Варіант | Складність | Час |
|---|---|---|---|
| Cisco CCAI Connector + CUCM NBR | Варіант 1 | Medium | 4–6 тижнів |
| Власний SRS-мікросервіс (Go/Rust) | Варіант 2 | High | 6–10 тижнів |
| OpenAI Realtime API інтеграція | Варіант 2 | Medium | 2–3 тижні |
| WebSocket Push Gateway + CRM UI | Варіант 2 | Medium | 3–5 тижнів |

**Ключові ризики:**
- **G.711 8kHz**: стандартний codec CUCM. Перевірити підтримку G.722 16kHz — покращує WER STT
- **RTP port pool**: планування пулу UDP-портів 16000–32000 для 7,500+ потоків

## Рекомендації для пілоту

**Варіант 1 (Google CCAI Connector)** — рекомендований для пілоту: мінімальна вартість розробки ($10–20K), офіційна підтримка Cisco + Google, найшвидший старт.

**Варіант 2 (OpenAI Realtime)** — розглядати після пілоту, якщо якість підказок Google Agent Assist недостатня для специфіки НП або потрібна власна CRM-інтеграція поза Finesse.

**Критерії успіху пілоту:**
- SIPREC-форкування не впливає на якість основного дзвінка
- Перші 100мс аудіо отримано < 200мс після старту дзвінка
- CPU SRS < 50% при 100 concurrent сесіях
- Точність транскрипції uk-UA > 85% WER на тестовій вибірці НП
